# Architecture Template — Full Example

This is a complete example of a well-written ARCHITECTURE.md for a real multi-repo system
(the PromptCloud.AI Salesforce automation platform). Use this as a reference for format,
depth, and writing style when generating architecture docs.

---

```markdown
# Architecture — langgraph

> Generated by arch-sync. Run `/arch-sync:sync` to regenerate.
> Last updated: 2026-02-20

## Overview

The `langgraph` repo is the AI orchestration layer of the Salesforce automation platform. It runs a multi-agent LangGraph graph (Supervisor + Sales/Service Cloud sub-agents) deployed on LangGraph Platform, classifying user requests as READ or CUD, executing read-only queries directly via MCP tools, and routing write operations through a human-in-the-loop approval flow before execution.

## System Components

| Component | Role | Tech | Deployed On |
|---|---|---|---|
| **langgraph** (this repo) | Multi-agent orchestration: intent classification, planning, approval gate, MCP tool execution | Python, LangGraph, LangChain, OpenAI | LangGraph Platform |
| **mcp** | 29 Salesforce operation tools (query, DML, describe, metadata) exposed via MCP/SSE | TypeScript, Express, jsforce | Railway |
| **salesforce-ai-automator** | React chat UI with plan approval, Salesforce OAuth, real-time SSE streaming | React, TypeScript, Vite, Supabase | Vercel / Lovable |
| **ai-agent-context-sync** | Periodic org data sync: calls MCP → chunks → embeds → stores in Supabase vector store | Python, FastAPI, LangChain | Railway (cron) |

## Inbound — Who Depends on This Repo

### salesforce-ai-automator (UI)
- **What it gets**: AI chat responses streamed in real-time; interrupt payloads when a plan needs approval
- **Protocol**: LangGraph Platform REST API + SSE streaming
- **Auth**: LangGraph API key in `x-api-key` header; requests routed via Supabase Edge Function `langgraph-chat`
- **Key endpoints**:
  - `POST /threads` — create conversation thread
  - `POST /threads/{thread_id}/runs/stream` — stream a run (SSE)
  - `POST /threads/{thread_id}/runs/{run_id}/command` — resume after interrupt (approve/edit/reject)
  - `GET /threads/{thread_id}/state` — read current thread state

## Outbound — What This Repo Depends On

### mcp (Salesforce MCP Server)
- **What we get**: 29 Salesforce operation tools (SOQL queries, DML, object describe, metadata management, Apex execution)
- **Protocol**: MCP over SSE (`MultiServerMCPClient` from `langchain-mcp-adapters`)
- **Auth**: No direct auth to MCP — we pass `userId` in every tool call argument; MCP fetches OAuth tokens from Supabase Vault using that userId
- **Key connection**: `MCP_SERVER_URL` env var → SSE endpoint (`/sse`)
- **Loading**: Tools loaded once at module import time via `get_mcp_tools_sync()`, gracefully falls back to `[]`

### Supabase (Vector Store)
- **What we get**: RAG context via similarity search on `daily_sync_documents`, `sales_cloud_documents`, `service_cloud_documents`
- **Protocol**: Supabase Python client (`supabase` package) + pgvector RPC functions
- **Auth**: `SUPABASE_SERVICE_ROLE_KEY` — service role, server-side only
- **Key tables**:
  - `daily_sync_documents` — org schema/config data synced by `ai-agent-context-sync`
  - `sales_cloud_documents` — results of Sales Cloud MCP tool calls (stored per-request)
  - `service_cloud_documents` — results of Service Cloud MCP tool calls

### OpenAI
- **What we get**: LLM completions (Supervisor, execution agents) and embeddings (vector store storage)
- **Protocol**: OpenAI REST API via `langchain-openai`
- **Auth**: `OPENAI_API_KEY`
- **Models**: `GPT_MODEL` (gpt-5) for Supervisor; `GPT_MODEL_MINI` (gpt-5-mini) for Planners + Execution Agents

## Data Flows

### Flow: User sends a read-only query ("Show me all fields on Account")

```
salesforce-ai-automator (UI)
  → Supabase Edge Function (langgraph-chat): forwards message + userId + orgId
  → LangGraph Platform: POST /threads/{id}/runs/stream
  → [this repo] thread_init: loads org context from Supabase daily_sync_documents
  → [this repo] supervisor: detects READ keywords, routes to sales_cloud agent
  → [this repo] sales_cloud agent: binds MCP tools, calls describe_object via mcp server
  → mcp server: fetches Salesforce OAuth token from Supabase Vault (using userId), calls Salesforce REST API
  → [this repo] tool_node: stores MCP result in sales_cloud_documents vector store
  → [this repo] sales_cloud agent: formats result, returns to END
  → SSE stream: response streamed back to UI
```

### Flow: User requests a write operation ("Create a Budget field on Account")

```
salesforce-ai-automator (UI)
  → [this repo] supervisor: detects CUD keywords, gathers org context via MCP tools
  → [this repo] sales_cloud_planner: reads tool results from context, generates 4-5 step plan
  → [this repo] human_approval: interrupt() — pauses execution
  → SSE stream: __interrupt__ event sent to UI with plan payload
salesforce-ai-automator (UI) — user clicks Approve
  → POST /threads/{id}/runs/{run_id}/command with {"resume": {"action": "approve"}}
  → [this repo] dispatch_execution: routes to sales_cloud agent
  → [this repo] sales_cloud agent: executes plan step-by-step via MCP tools
  → mcp server: performs Salesforce DML operations
  → SSE stream: results streamed to UI
```

### Flow: Daily context sync (background job)

```
ai-agent-context-sync (Railway cron)
  → mcp server: calls sync_full_org MCP tool with userId
  → mcp server: fetches full org data from Salesforce
  → ai-agent-context-sync: chunks response (800 chars, 200 overlap)
  → ai-agent-context-sync: generates embeddings via OpenAI text-embedding-3-small
  → Supabase: upserts to daily_sync_documents table
  → [this repo] thread_init (next session): retrieves top-k chunks for org context
```

## Integration Contracts

### Authentication & Credentials

**Pattern:** Never pass credentials directly. Always use userId for vault lookup.

- **This repo → MCP**: We pass `userId` (UUID) in every MCP tool call argument. MCP fetches the user's Salesforce OAuth tokens from Supabase Vault using that userId. We never hold or see the OAuth tokens.
- **UI → This repo**: UI sends `userId` in the LangGraph run input state. We preserve it in `state.user_id` and re-attach it after sub-agent invocations (`result["user_id"] = user_id`).
- **This repo → Supabase**: Service role key in env. No per-user auth — we use userId as a filter on all vector store queries for tenant isolation.

### Shared Data Schemas

**LangGraph Interrupt Payload** (sent to UI on plan approval):
```json
{
  "type": "plan_approval",
  "question": "Please review the execution plan and choose an action:",
  "plan": "<plan text>",
  "options": {
    "approve": "Approve the plan and proceed with execution",
    "edit": "Request changes to the plan (provide feedback)",
    "reject": "Reject the plan and cancel execution"
  }
}
```

**Resume Command** (sent by UI after user decision):
```json
{ "resume": { "action": "approve" } }
{ "resume": { "action": "edit", "feedback": "Change field type to Picklist" } }
{ "resume": { "action": "reject" } }
```

**LangGraph Run Input State** (sent by UI per message):
```json
{
  "messages": [{ "type": "human", "content": "<user message>" }],
  "user_id": "<supabase-user-uuid>",
  "org_id": "<salesforce-org-id>"
}
```

### Protocols & Transport

| Integration | Protocol | Details |
|---|---|---|
| UI → This repo | LangGraph Platform REST + SSE | `POST /runs/stream` for streaming, `POST /command` for resumes |
| This repo → MCP | MCP over SSE | `langchain-mcp-adapters` MultiServerMCPClient, SSE transport |
| This repo → Supabase | Supabase Python client | pgvector RPC calls (`match_*` functions) |
| This repo → OpenAI | OpenAI REST API | Via langchain-openai, streaming completions |

### Shared Environment Variables

These must be consistent across all components that use them:

| Variable | Shared With | Purpose |
|---|---|---|
| `SUPABASE_URL` | All components | Shared Supabase project URL |
| `SUPABASE_SERVICE_ROLE_KEY` | langgraph, ai-agent-context-sync | Server-side DB access |
| `MCP_SERVER_URL` | langgraph, ai-agent-context-sync | MCP server SSE endpoint |
| `OPENAI_API_KEY` | langgraph, ai-agent-context-sync | OpenAI API access |
| `EMBEDDING_MODEL` | langgraph, ai-agent-context-sync | Must match for vector search to work (`text-embedding-3-small`) |
| `EMBEDDING_DIMENSIONS` | langgraph, ai-agent-context-sync | Must match (`1536`) |
```

## Notes on This Example

**What makes this a good architecture doc:**

1. **Specific over generic** — Says "pgvector RPC calls (`match_*` functions)" not just "database queries"
2. **Perspective maintained** — Every section is from `langgraph`'s viewpoint; other repos described in terms of how they relate to `langgraph`
3. **Real names** — Uses actual tool names (`get_mcp_tools_sync`, `MultiServerMCPClient`), actual env var names, actual endpoint paths
4. **Flows are concrete** — Each arrow shows a real component making a real call; includes what data moves
5. **Contracts are actionable** — The auth section tells a new developer exactly what to implement; the schema section has real JSON
6. **Honest about complexity** — The interrupt/resume flow shows all the steps including the UI side, making it clear how the whole loop works
