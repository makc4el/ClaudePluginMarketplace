# System Architecture Analysis

Generates a repo-specific ARCHITECTURE.md by synthesizing CLAUDE.md, INTEGRATION.md, and README.md files collected from sibling repos into a unified system view.

## Purpose

When working in a single repo of a multi-platform system, Claude lacks visibility into how other components work and how they connect. This skill synthesizes documentation from all sibling repos into a unified architecture view from the current repo's perspective, covering:
- What depends on this repo (inbound)
- What this repo depends on (outbound)
- End-to-end data flows through the system
- Shared integration contracts (auth, schemas, protocols)

## Analysis Process

### Step 1: Identify the Current Repo's Role

Read the current repo's own CLAUDE.md (if it exists) to understand:
- What this repo does (its domain, responsibility, deployment target)
- Its primary interfaces (API it exposes, events it emits, files it produces)
- Tech stack and runtime environment

If CLAUDE.md is missing, infer the role from directory name and README.

### Step 2: Read Each Sibling Repo's Docs

For each repo cached in `.claude/system-context/<repo-name>/`:

**From CLAUDE.md** — extract:
- What the component does (1-sentence role)
- Its deployment target and tech stack
- Interfaces it exposes (API endpoints, SSE streams, MCP tools, DB tables, etc.)
- Interfaces it consumes (what it calls out to)
- Any shared state or data contracts it maintains

**From INTEGRATION.md** — extract:
- Exact API shapes: endpoints, request/response bodies, auth headers
- Stream formats (SSE event types, payloads)
- Protocol details (REST, MCP/SSE, gRPC, webhooks)
- Authentication flows and token formats
- Error codes and retry behavior
- Any SDK or client examples

**From README.md** — extract:
- High-level purpose (useful for the Components table)
- Deployment environment
- External dependencies (databases, third-party APIs, cloud services)

### Step 3: Map Dependencies

For each sibling repo, determine:

**Inbound to current repo** (they call/read us):
- Does this sibling reference the current repo's API, tools, or data?
- What does it request from us? Which endpoints or tools?
- How does it authenticate?

**Outbound from current repo** (we call/read them):
- Does the current repo's documentation mention calling this sibling?
- What do we request? Which endpoints or tools?
- How do we authenticate?

**Shared contracts** (both use):
- Same Supabase project, tables, or vault?
- Same MCP server URL?
- Same auth token format?
- Shared environment variables?

### Step 4: Trace Data Flows

Identify 2-5 significant flows that involve the current repo. A flow is a sequence of calls that produce a meaningful result for the end user (e.g., "User sends message → receives Salesforce data").

For each flow:
1. Name it after the user outcome ("User sends chat message", "Context sync runs")
2. Trace it from the component that initiates it to the final result
3. Mark where the current repo appears in the chain and what it does there
4. Show the path with arrows: `A → B → C → result`

Focus on flows where this repo plays an **active role**, not just passes-through.

## ARCHITECTURE.md Structure

Generate the file using this structure. Adapt sections based on what's actually true — omit sections that don't apply (e.g., no "Inbound" section if nothing depends on this repo).

## The Generated File Rule

**ARCHITECTURE.md is a generated file. Never edit it directly.**

It will be fully overwritten the next time `/arch-sync:sync` runs. Any manual edits are silently lost.

The correct workflow when you discover something wrong or missing:

1. Identify which source doc owns the information:
   - Something about **this repo's internals** → update this repo's `CLAUDE.md`
   - Something about **a sibling repo's API contract** → update that repo's `INTEGRATION.md`
   - Something **shared across repos** → update whichever repo's doc is the authoritative source
2. Save the source doc
3. Re-run `/arch-sync:sync` to regenerate `ARCHITECTURE.md`

**When cross-referencing docs against actual code and finding a discrepancy** (e.g., docs say the API uses key `"approval"` but the code reads key `"action"`):
- Do **not** patch ARCHITECTURE.md
- Note the discrepancy in the relevant `CLAUDE.md` or `INTEGRATION.md` with a clear `⚠️` marker
- Re-run sync so the warning propagates into the generated file automatically

## Writing Guidelines

**Be specific, not generic.** Replace "Component A calls Component B" with the actual endpoint or tool name. Replace "uses authentication" with the actual mechanism ("Supabase Vault lookup using userId parameter").

**Perspective matters.** Every sentence should be written from the current repo's viewpoint. When describing other components, describe them in terms of how they relate to *this* repo.

**Use actual names.** Use the real names of components, endpoints, tools, and environment variables from the scanned documentation. Do not generalize to "the API" or "the service".

**Flows must be concrete.** Each flow should name the actual components and show the actual sequence, not a generic template.

**Keep Overview short.** 2-3 sentences maximum. The rest of the file provides depth.

## Handling Missing Documentation

When a sibling repo's files are missing or sparse:
- Note in the Components table: "Role inferred from repo name"
- In Inbound/Outbound: describe what is known from the current repo's docs (e.g., if CLAUDE.md mentions calling the MCP server, document that even if the MCP server's CLAUDE.md wasn't scanned)
- Add a note in the relevant section: "⚠️ INTEGRATION.md not available — details based on current repo's perspective only"
